# Profiles UI

## Scope Snapshot

- In scope: React-приложение для управления профилями Content Shield, встроенное в gateway как статика; страницы списка, деталей и формы профиля с inline-редакторами словарей и препроцессоров.
- Out of scope: UI для логов/инцидентов Shield; отдельный Docker-образ для фронтенда; dashboard реального времени.

## Цель

Оператор Content Shield получает веб-интерфейс для CRUD-управления профилями справочников: просмотр списка, создание/редактирование профиля с настройкой детекторов, реакций, словарей и препроцессоров. UI встраивается в бинарник gateway через Go embed и раздаётся тем же HTTP-сервером — без отдельного образа или反 proxy.

## Основной сценарий

1. Пользователь открывает `/profiles` в браузере — видит таблицу всех профилей с slug, названием, статусом.
2. Нажимает «Создать профиль» — открывается форма с разделами: общие поля, детекторы, реакции, словари, препроцессоры.
3. Заполняет форму, сохраняет — POST /api/v1/profiles, после успеха редирект на `/profiles/:slug`.
4. В деталях профиля доступны inline-редакторы: словари (entries key-value), препроцессоры (CSV/JSON rules).
5. Ошибка валидации — сообщение в форме, данные не теряются.

## User Stories

- P1 Story: Оператор видит список профилей с основными полями и может открыть карточку профиля.
- P1 Story: Оператор может создать новый профиль через форму с заполнением имени, slug, описания, тегов и набора детекторов/реакций.
- P2 Story: Оператор может редактировать словари профиля (inline-менеджер entries) и препроцессоры (правила CSV/JSON) внутри формы профиля.
- P2 Story: Оператор может удалить профиль из карточки или списка.

## MVP Slice

Таблица профилей + форма создания с базовыми полями и фиксированными детекторами/реакциями (без inline-редакторов словарей и препроцессоров). AC-001, AC-002, AC-005.

## First Deployable Outcome

После первого implementation pass: `npm run build` в `ui/` собирает статику, `go build` встраивает её через `embed.FS`, на `GET /profiles*` отдаётся SPA. Форма создания профиля отправляет корректный POST и видит успех/ошибку.

## Scope

1. Vite + React + TypeScript проект в `ui/` с роутингом react-router-dom.
2. Три страницы: ProfileList (`/profiles`), ProfileDetail (`/profiles/:slug`), ProfileForm (`/profiles/new` и `/profiles/:slug/edit`).
3. Набор UI-компонентов для конфигурации профиля: DetectorConfigurator, ReactionSelector, PatternEditor, DictionaryEditor, PreprocessorEditor.
4. Словари и препроцессоры редактируются inline внутри формы профиля (модальные/раскрывающиеся панели), а не на отдельных страницах.
5. API-клиент для `/api/v1/profiles/*`.
6. Сборка статики (`vite build` в `ui/dist/`) и встраивание в Go-бинарник gateway через `//go:embed`.
7. Gin `NoRoute` handler для SPA: статические файлы (`/assets/*`) сначала ищутся в `ui/dist/`, не найденные пути отдают `index.html`.
8. Makefile-таргеты: `ui-build`, `ui-dev` (Vite HMR proxy на Gin-порт).

## Контекст

- Go-бинарник — единственный артефакт развёртывания; никакого отдельного reverse proxy/nginx для статики.
- dev-режим: Vite dev server с proxy на `localhost:<gateway-port>` для API-запросов.
- Текущий gateway уже имеет API-маршруты `/api/v1/profiles/*` (задача 40-profiles-api).
- Dockerfile должен быть расширен для двухстадийной сборки: node → go build.
- React Router использует BrowserRouter; Gin обрабатывает fallback через `NoRoute` (catch-all), зарегистрированный после всех API-роутов.
- API пагинация через query-параметры `?page=&page_size=` — клиент передаёт, API возвращает `{data, total, page, page_size}`.

## Зависимости

- `40-profiles-api` — REST API `/api/v1/profiles/*` (CRUD профилей, PATCH словаря).
- React 18+, react-router-dom v6, Vite 5+, TypeScript 5+.
- `embed` из stdlib Go для встраивания статики.

## Требования

- RQ-001 Система ДОЛЖНА отдавать SPA на всех GET-запросах без префикса `/api/`, `/health`, `/ready`, `/live`.
- RQ-002 Пользователь ДОЛЖЕН видеть список профилей (slug, name, статус, теги) с пагинацией через query-параметры `?page=&page_size=`, фильтрацией по имени/тегам.
- RQ-003 Пользователь ДОЛЖЕН создавать профиль через форму с полями: name, slug, description, tags, detectors, reactions.
- RQ-004 Пользователь ДОЛЖЕН просматривать детали профиля: все поля плюс связанные словари и препроцессоры.
- RQ-005 Пользователь ДОЛЖЕН редактировать словари профиля (entries key-value) inline в форме профиля без перехода на другую страницу.
- RQ-006 Пользователь ДОЛЖЕН редактировать препроцессоры профиля (CSV/JSON rule builder) inline в форме профиля.
- RQ-007 Сборка `vite build` ДОЛЖНА быть частью `docker-build` (multi-stage: node → go).
- RQ-008 Make target `ui-dev` ДОЛЖЕН запускать Vite dev server с proxy API-запросов на Gin.
- RQ-009 При ошибке валидации формы данные не ДОЛЖНЫ сбрасываться, сообщение об ошибке ДОЛЖНО быть показано.

## Вне scope

- UI для просмотра логов и инцидентов Shield (отдельная фича).
- Dashboard реального времени (metrics, трафик).
- Envoy-режим и его UI.
- RBAC/авторизация в UI (роли пока не реализованы в API).
- Экспорт/импорт профилей (отдельная фича).
- Отдельный Docker-образ для фронтенда — UI всегда встраивается в gateway.

## Критерии приемки

### AC-001 SPA отдаётся gateway на корневых маршрутах

- Почему это важно: браузер должен получать index.html для SPA-роутинга.
- **Given** запущенный gateway со встроенной статикой
- **When** браузер запрашивает `GET /profiles` или `GET /profiles/new`
- **Then** сервер возвращает HTML `index.html` (Content-Type: text/html), а не 404
- Evidence: `curl -s -o /dev/null -w "%{http_code}" http://localhost:<port>/profiles` возвращает 200, заголовок Content-Type: text/html

### AC-002 Список профилей загружается с API с пагинацией

- Почему это важно: пользователь видит актуальные данные с навигацией по страницам.
- **Given** открыта страница `/profiles` в браузере
- **When** страница загружается
- **Then** отправляется GET `/api/v1/profiles?page=1&page_size=20`, ответ содержит `{data, total, page, page_size}`, рендерится таблица с пагинацией
- Evidence: в DevTools виден XHR-запрос `/api/v1/profiles?page=1&page_size=20` со статусом 200; таблица содержит строки; при наличии >1 страницы отображаются кнопки пагинации

### AC-003 Создание профиля через форму

- Почему это важно: оператор может добавить новый профиль.
- **Given** открыта страница `/profiles/new`
- **When** пользователь заполняет обязательные поля (name, slug) и нажимает «Сохранить»
- **Then** отправляется POST `/api/v1/profiles`; при успехе (201) происходит редирект на `/profiles/:slug`
- Evidence: после сохранения URL меняется на `/profiles/:slug`; профиль отображается в списке

### AC-004 Редактирование профиля

- Почему это важно: оператор может изменить существующий профиль.
- **Given** открыта страница `/profiles/:slug`
- **When** пользователь нажимает «Редактировать», меняет поля и сохраняет
- **Then** отправляется PUT `/api/v1/profiles/:slug`; при успехе обновлённые поля отображаются на странице деталей
- Evidence: после сохранения имя/описание профиля обновлены на странице деталей

### AC-005 Валидация формы с сохранением данных

- Почему это важно: UX без потери введённого.
- **Given** открыта форма создания/редактирования профиля
- **When** пользователь вводит некорректный slug (например, с пробелами) и нажимает «Сохранить»
- **Then** форма не отправляется на сервер (или сервер возвращает 422), ошибка отображается в UI, все введённые данные сохранены
- Evidence: после ошибки поля формы содержат те же значения, что до отправки

### AC-006 Inline-редактор словарей в форме профиля

- Почему это важно: оператор управляет словарём без перехода на другую страницу.
- **Given** открыта форма редактирования профиля
- **When** пользователь открывает секцию «Словари» и добавляет entry (key=value)
- **Then** entry отображается в списке; при сохранении профиля словарь отправляется на сервер
- Evidence: после сохранения и повторного открытия профиля добавленная entry присутствует

### AC-007 Inline-редактор препроцессоров в форме профиля

- Почему это важно: оператор настраивает преобразование данных внутри формы.
- **Given** открыта форма редактирования профиля
- **When** пользователь открывает секцию «Препроцессоры» и добавляет CSV-правило
- **Then** правило отображается в списке; при сохранении препроцессоры отправляются на сервер
- Evidence: после сохранения и повторного открытия профиля добавленное правило присутствует

### AC-008 Dev-режим: Vite HMR + API proxy

- Почему это важно: разработчик не собирает статику при каждой правке.
- **Given** запущен `make ui-dev` (Vite dev server) и gateway
- **When** разработчик открывает `http://localhost:5173/profiles`
- **Then** страница загружается, API-запросы проксируются на `localhost:<gateway-port>`
- Evidence: в DevTools XHR-запрос `/api/v1/profiles` уходит на `localhost:5173` и успешно проксируется на gateway

### AC-009 Docker-сборка включает UI

- Почему это важно: единый образ для деплоя.
- **Given** выполнен `make docker-build`
- **When** контейнер запущен и открыт `http://localhost:<port>/profiles`
- **Then** SPA отдаётся как статика из бинарника
- Evidence: `docker build` проходит успешно; `curl localhost:<port>/profiles` возвращает HTML

### AC-010 Удаление профиля из UI

- Почему это важно: оператор может удалить ненужный профиль.
- **Given** открыта страница деталей профиля `/profiles/:slug`
- **When** пользователь нажимает «Удалить» и подтверждает действие
- **Then** отправляется DELETE `/api/v1/profiles/:slug`; после успеха редирект на `/profiles` с уведомлением
- Evidence: профиль исчезает из списка; уведомление об успешном удалении отображается

### AC-011 Состояния загрузки и пустой список

- Почему это важно: пользователь не видит пустой/сломанный экран.
- **Given** открыта страница `/profiles`
- **When** данные загружаются (или API вернул пустой массив)
- **Then** отображается индикатор загрузки; при пустом результате — сообщение «Нет профилей. Создайте первый профиль»
- Evidence: визуальный спиннер/скелетон во время запроса; сообщение о пустом списке при 200 с `[]`

## Допущения

- API `/api/v1/profiles/*` полностью реализовано в задаче 40-profiles-api и стабильно.
- Словари передаются как массив `{key, value}` в теле профиля; PATCH словаря через отдельный endpoint уже есть.
- Препроцессоры передаются как массив правил `{type, config}` в теле профиля.
- Gin-сервер уже имеет CORS middleware, необходимый для dev-режима Vite.
- React Router использует BrowserRouter; Gin обрабатывает fallback через `NoRoute`.
- Node.js 20+ доступен в CI/CD для сборки статики.

## Критерии успеха

- SC-001 Страница списка профилей загружается и рендерится менее чем за 2 секунды (LCP) при 50 профилях.
- SC-002 Сборка `docker-build` с UI занимает не более 3 минут.

## Краевые случаи

- Пустой список профилей: сообщение с CTA «Создать первый профиль».
- Ошибка сети: тост/баннер «Не удалось загрузить профили. Попробуйте позже».
- 404 профиль: на `/profiles/non-existent-slug` показать «Профиль не найден» с кнопкой «К списку».
- Конфликт slug (409): показать сообщение «Профиль с таким slug уже существует».
- Очень длинные списки entries/правил: виртуализация списка или scroll.
- Браузерный back/forward: корректное восстановление состояния роута.

## Открытые вопросы
