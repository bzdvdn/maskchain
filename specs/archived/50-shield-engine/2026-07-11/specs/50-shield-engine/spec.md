# Shield Engine — Orchestrator сканирования

## Scope Snapshot

- In scope: оркестратор, который по запросу (текст + профиль) выстраивает цепочку препроцессоры → словари → детекторы → PolicyEvaluator → ReactionExecutor и возвращает результат.
- Out of scope: реализация отдельных детекторов, препроцессоров, словарей, реакций — все они уже существуют как domain-компоненты.

## Цель

Разработчик интеграции получает единый публичный API `ShieldEngine.Scan(ctx, ScanRequest) ScanResult`, который скрывает всю сложность построения пайплайна из конфигурации профиля. Успех фичи измеряется тем, что интеграционный тест, прогоняющий реальные данные через полную цепочку (препроцессор → словарь → детектор → реакция), завершается с корректным ScanResult.

## Основной сценарий

1. Клиент вызывает `ShieldEngine.Scan(ctx, ScanRequest{Text, ProfileSlug})`.
2. `ShieldEngine` загружает профиль по `ProfileSlug` через `ProfileRepository`.
3. `ScanPipelineFactory` строит цепочку обработки из конфигурации профиля: препроцессоры, словарные детекторы, regex-детекторы.
4. Каждый препроцессор (CSV/JSON) маскирует структурированные данные, заменяя значения на placeholders вида `{{csv.ns.N}}`, `{{json.ns.N}}`.
5. Словари (wordlist matching) и детекторы (PII, secrets, financial, PHI)扫描ят обработанный текст.
6. `PolicyEvaluator` определяет реакцию по приоритету среди найденных инцидентов.
7. `ReactionPipeline` выполняет реакцию (block/redact/mask/alert).
8. В случае mask-реакции финальный текст содержит placeholders, карта подстановок возвращается в результате.
9. `ShieldEngine` возвращает `ScanResult` со статусом, инцидентами и (опционально) обработанным текстом.

## User Stories

- P1 Story: Как разработчик, я хочу вызвать `ShieldEngine.Scan` с текстом и profile slug, чтобы получить результат сканирования с реакцией, не вникая в устройство пайплайна.
- P2 Story: Как разработчик, я хочу вызвать `ApplyPolicyUseCase` на уже существующем `ScanResult`, чтобы переоценить политику без повторного сканирования.

## MVP Slice

- `ShieldEngine.Scan` с полным пайплайном: препроцессоры + словари + детекторы + PolicyEvaluator + ReactionPipeline.
- Placeholder-based masking для CSV и JSON.
- Интеграционный тест, покрывающий полную цепочку.

## First Deployable Outcome

Один интеграционный тест (Go test), который создаёт профиль с CSV-препроцессором, словарём, PII-детектором и реакцией redact, прогоняет текст через `ShieldEngine.Scan` и проверяет статус, наличие инцидентов и корректность placeholders.

## Scope

- `ShieldEngine` — public API: `Scan(ctx, req ScanRequest) ScanResult`.
- `ScanUseCase` — реализация оркестрации в `src/internal/app/usecase/shield/`.
- `ApplyPolicyUseCase` — use case для применения PolicyEvaluator к готовому ScanResult.
- `ScanPipelineFactory` — построение цепочки обработки из конфигурации Profile.
- Placeholder schema: `{{csv.<ns>.<N>}}`, `{{json.<ns>.<N>}}`, `{{p.<ns>.<N>}}`, `{{dict.<ns>.<N>}}`.
- Интеграционные тесты полного пайплайна.
- Новые директории: `src/internal/app/usecase/shield/`.

## Контекст

- Детекторы, препроцессоры, словари и реакции уже реализованы в `src/internal/domain/shield/` — оркестратор их использует, не дублирует.
- `Profile` уже содержит `Detectors`, `Dictionaries`, `Preprocessors` поля.
- `service.ScanPipeline` (domain) сканирует только детекторами — не использует препроцессоры и словари. Новый `ScanPipelineFactory` будет работать на app-уровне.
- Placeholder-based masking уже реализован в препроцессорах (CSVProcessor, JSONProcessor) — формат расширяется на словари и детекторы через единую схему.

## Зависимости

- Зависит от `ProfileRepository` для загрузки профиля.
- Использует `DetectorRegistry`, `DictionaryRepository`, препроцессоры и реакции из domain.
- `none` внешних сервисных зависимостей.

## Требования

- RQ-001 `ShieldEngine.Scan` ДОЛЖЕН загружать профиль по `ProfileSlug`, строить пайплайн и выполнять полный цикл обработки.
- RQ-002 `ScanPipelineFactory` ДОЛЖЕН строить цепочку из препроцессоров, словарных детекторов и regex-детекторов в указанном порядке на основе конфигурации профиля.
- RQ-003 Система ДОЛЖНА поддерживать placeholders вида `{{csv.<ns>.<N>}}`, `{{json.<ns>.<N>}}`, `{{p.<ns>.<N>}}`, `{{dict.<ns>.<N>}}` для всех этапов пайплайна.
- RQ-004 `ApplyPolicyUseCase` ДОЛЖЕН применять `PolicyEvaluator` к существующему `ScanResult` и возвращать выбранную реакцию без повторного сканирования.
- RQ-005 Если профиль не найден, `ShieldEngine.Scan` ДОЛЖЕН возвращать ошибку с указанием неизвестного slug.
- RQ-006 Если в пайплайне нет ни одного активного детектора/словаря, `ShieldEngine.Scan` ДОЛЖЕН возвращать статус `clean` без выполнения реакций.

## Вне scope

- Реализация новых типов детекторов, препроцессоров, словарей или реакций.
- Изменение существующих domain-интерфейсов Detector, Processor, ReactionExecutor.
- UI-изменения для настройки shield engine.
- Мониторинг, метрики, аудит-логи вызовов ShieldEngine (отдельная фича).
- Кэширование результатов сканирования.

## Критерии приемки

### AC-001 Полный пайплайн с препроцессором, словарём и детектором

- Почему это важно: гарантирует, что оркестратор корректно собирает и выполняет все этапы.
- **Given** профиль с CSV-препроцессором (маскировать колонку `email`), словарём (точное совпадение `secret123`), PII-детектором и реакцией redact
- **When** `ShieldEngine.Scan` вызывается с текстом, содержащим CSV-строку и `secret123`
- **Then** результат содержит статус `suspicious`, инциденты от словаря и PII-детектора, текст с применённым redact
- Evidence: интеграционный тест проверяет `ScanResult.Status`, `len(Incidents) >= 2`, финальный текст не содержит исходных sensitive-данных

### AC-002 Placeholder-based masking в результате

- Почему это важно: вызывающая сторона должна иметь возможность восстановить исходные данные по карте подстановок.
- **Given** профиль с препроцессором, заменяющим значения на placeholders
- **When** `ShieldEngine.Scan` завершает обработку с mask-реакцией
- **Then** результат содержит мапу `Replacements` с соответствием placeholder → исходное значение
- Evidence: тест проверяет, что `Replacements` содержит ключи вида `{{csv.default.0}}` и восстановление текста по мапе даёт исходную строку

### AC-003 ApplyPolicyUseCase без повторного сканирования

- Почему это важно: позволяет переоценить реакцию для уже проверенного текста без повторного прогона всего пайплайна.
- **Given** существующий `ScanResult` с инцидентами
- **When** вызывается `ApplyPolicyUseCase.Execute(ctx, scanResult)`
- **Then** возвращается выбранная реакция на основе приоритета severity среди инцидентов
- Evidence: unit-тест, создающий ScanResult с тремя инцидентами разной severity, проверяет, что реакция соответствует highest severity

### AC-004 Профиль не найден

- Почему это важно: клиент должен получить детерминированную ошибку `ErrProfileNotFound`, а не панику или некорректный результат.
- **Given** несуществующий `ProfileSlug`
- **When** `ShieldEngine.Scan` вызывается с этим slug
- **Then** возвращается ошибка `ErrProfileNotFound`
- Evidence: тест с несуществующим slug проверяет `errors.Is(err, ErrProfileNotFound)`

### AC-005 Пустой пайплайн (нет активных детекторов/словарей)

- Почему это важно: профиль может быть временно отключён, система должна корректно обрабатывать этот случай.
- **Given** профиль без активных детекторов и словарей
- **When** `ShieldEngine.Scan` вызывается с любым текстом
- **Then** результат содержит статус `clean` и пустой список инцидентов
- Evidence: тест проверяет `ScanResult.Status == clean` и `len(Incidents) == 0`

### AC-006 Placeholder-форматы для всех этапов

- Почему это важно: единая схема placeholders обеспечивает предсказуемость для вызывающей стороны.
- **Given** профиль, включающий CSV-препроцессор, JSON-препроцессор, PII-детектор и словарь
- **When** `ShieldEngine.Scan` обрабатывает текст, содержащий CSV, JSON, PII и словарные совпадения
- **Then** placeholders для CSV имеют вид `{{csv.<ns>.<N>}}`, для JSON — `{{json.<ns>.<N>}}`, для детекторов — `{{p.<ns>.<N>}}`, для словарей — `{{dict.<ns>.<N>}}`
- Evidence: тест проверяет, что финальный текст содержит все четыре формата placeholders в соответствующих позициях

## Допущения

- `ProfileRepository.Load` доступен и возвращает профиль с заполненными полями Preprocessors, Dictionaries, Detectors.
- Все детекторы, препроцессоры и реакции thread-safe и могут быть переиспользованы между вызовами.
- Порядок этапов фиксирован: препроцессоры → словари → детекторы → PolicyEvaluator → ReactionExecutor.

## Критерии успеха

- SC-001 Время полного цикла сканирования (текст 1KB, профиль с 1 препроцессором + 3 детекторами) < 100ms.
- SC-002 Интеграционный тест выполняется за < 2s в CI.

## Краевые случаи

- Пустой текст на входе — результат `clean`, 0 инцидентов.
- Профиль отключён (enabled=false) — ShieldEngine возвращает ошибку или `clean` (определить на plan).
- Дублирующиеся инциденты от разных детекторов на один и тот же фрагмент — дедупликация не требуется в MVP.
- Препроцессор не может обработать формат — ошибка логируется, пайплайн прерывается.
