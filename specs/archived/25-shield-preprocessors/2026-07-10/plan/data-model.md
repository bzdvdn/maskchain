# Shield Preprocessors: CSV/JSON Masking Pipeline — Модель данных

## Scope

- Связанные `AC-*`: `AC-007`, `AC-008`
- Связанные `DEC-*`: `DEC-001`
- Статус: `changed`

## Сущности

### DM-001 Profile (расширение)

- Назначение: конфигурация Content Shield для tenant'а. Добавляется поле `preprocessors` — список препроцессоров, встроенных в профиль.
- Источник истины: таблица `profiles` в PostgreSQL, JSONB-колонка `preprocessors`
- Инварианты: элемент массива `preprocessors` обязан содержать валидный `type` (`csv` или `json`) и непустой `name`
- Связанные `AC-*`: `AC-008`
- Связанные `DEC-*`: `DEC-001`
- Поля (новые):
  - `preprocessors` — `[]PreprocessorDef`, optional, default `nil`/`[]`, JSONB в PG
- Жизненный цикл: без изменений — поле создаётся/обновляется вместе с профилем
- Замечания по консистентности: невалидные препроцессоры (неизвестный `type`) приводят к ошибке загрузки профиля

### DM-002 PreprocessorDef

- Назначение: декларативное описание одного препроцессора для сериализации в JSONB-поле профиля
- Источник истины: правило из профиля, не имеет собственного identity
- Инварианты:
  - `name` — непустая строка, уникальна в рамках одного профиля
  - `type` — одно из: `csv`, `json`
  - `rules` — непустой список, каждый rule валиден для своего типа
- Связанные `AC-*`: `AC-001`–`AC-007`
- Поля:
  - `name` — `string`, required, идентификатор препроцессора в профиле
  - `type` — `string`, required, одно из: `"csv"`, `"json"`
  - `rules` — `[]Rule`, required, правила маскировки
- Жизненный цикл: создаётся/редактируется/удаляется как часть профиля. Нет собственного lifecycle

### DM-003 Rule (вложенный в PreprocessorDef)

- Назначение: одно правило маскировки — какие поля/колонки и как маскировать
- Инварианты:
  - Для `type: "csv"`: `columns` непустой, `mask` — `full` или `surname`
  - Для `type: "json"`: `path` непустой валидный JSONPath, `mask` — `full`
- Поля:
  - `columns` — `[]string`, optional (только для CSV), имена колонок для маскировки
  - `path` — `string`, optional (только для JSON), JSONPath к полю
  - `mask` — `string`, required, `"full"` или `"surname"`
- Жизненный цикл: часть PreprocessorDef

### DM-004 ProcessResult

- Назначение: результат выполнения препроцессора — замаскированный текст и словарь замен
- Источник истины: runtime-значение, не персистится
- Поля:
  - `ModifiedText` — `string`, текст с плейсхолдерами
  - `Replacements` — `map[string]string`, оригинал → плейсхолдер
- Жизненный цикл: создаётся при каждом вызове `Process()`, живёт до завершения обработки запроса в Shield Engine

## Связи

- `DM-002 (PreprocessorDef) -> DM-003 (Rule)`: 1:N, composition (правила живут только в контексте препроцессора)
- `DM-001 (Profile) -> DM-002 (PreprocessorDef)`: 1:N, composition (препроцессоры живут только в профиле)
- `DM-004 (ProcessResult)`: не связан с persisted entities

## Производные правила

- JSONB-колонка `preprocessors` в PostgreSQL хранит `[]PreprocessorDef` как массив JSON-объектов. При загрузке профиля массив десериализуется в Go-структуры; ошибка десериализации = ошибка загрузки профиля.
- Пустой или NULL-массив означает «без препроцессоров» — pipeline не вызывает preprocessing step.

## Переходы состояний

- `preprocessors` не имеет собственных состояний — меняется атомарно при обновлении профиля

## Вне scope

- Индекс на JSONB-поле `preprocessors` — поиск по содержимому препроцессоров не требуется
- Shared-препроцессоры (отдельная таблица `preprocessors` со ссылкой из профиля)
- Entity для Rule как самостоятельной сущности
